# Sanity seed data

NDJSON snapshots of the initial content for each singleton / collection.
Load into a fresh dataset with the Sanity CLI:

```bash
npx sanity dataset import ./sanity/seed/profileData.ndjson production
```

## Files

- `profileData.ndjson` — Profile hero singleton (AC 7). Mirrors the legacy
  hardcoded values from `src/components/Profile/index.jsx`:
  name, title, openToWork toggle + label, multi-paragraph tagline,
  résumé URL + label.
- `skillGroups.ndjson` — Nine `skillGroup` documents (Sub-AC 8.2).
  Mirrors the legacy `skillGroups` array exported from
  `src/components/Skills/index.jsx`. Generated by
  `scripts/seed-skill-groups.mjs`. Each doc carries a stable
  `_id` (`skillGroup.<slug>`) and an `order` field on a 10/20/30… scale
  for GROQ `orderAsc` sort.
- `roleEntries.ndjson` — Three `roleEntry` documents (Sub-AC 6.3).
  Mirrors the legacy three-card grid in pre-redesign
  `src/components/Roles/index.jsx` (git 9f34352) and `data.js`
  (`toolNames`, `leaderExamples`). Generated by
  `scripts/seed-role-entries.mjs`. Each doc carries a stable
  `_id` (`roleEntry.<slug>`) and an `order` field on a 10/20/30 scale
  for GROQ `orderAsc` sort. Lossy note: `leaderExamples` hrefs are
  dropped — schema has no link field; titles preserved in `audience`.
- `footerData.ndjson` — `footerData` singleton (Sub-AC 6.4).
  Mirrors the legacy `linkData` array in `src/components/Footer/data.js`.
  Five social links (github, linkedin, rss, strava, mail) with stable
  `_key`s and an `order` field on a 10/20/30/40/50 scale for in-Studio
  reorder without re-numbering all entries. Stable doc `_id`
  (`footerData.singleton`) keeps re-imports idempotent. Generated by
  `scripts/seed-footer-data.mjs`. `copyright` left unset — legacy footer
  renders no copyright line.
- `workExperience.ndjson` — Seven `workExperienceEntry` documents
  (Sub-AC 6.2). Mirrors `src/components/WorkExperience/data.js`
  (Peloton → TuneIn). Each doc carries company, href, logo
  (uploaded via `_sanityAsset` from `src/assets/img/`), dates, title,
  description, tools[], and `order` (1 = newest, 7 = oldest) for
  newest-first GROQ `orderAsc` sort. Stable `_id`s
  (`workExperience.<slug>`) keep re-imports idempotent.

  Import (logos + docs in one shot):

  ```bash
  npx sanity dataset import ./sanity/seed/workExperience.ndjson production
  ```

  The `_sanityAsset: "image@file://../../src/assets/img/<file>"` paths
  resolve relative to the NDJSON file — the importer uploads each logo and
  rewrites the references before document insert.

## Regenerating

```bash
npm run seed:skill-groups                           # write NDJSON only
node scripts/seed-skill-groups.mjs --import         # write + import production
node scripts/seed-skill-groups.mjs --import --dataset=staging

npm run seed:role-entries                           # write NDJSON only
node scripts/seed-role-entries.mjs --import         # write + import production
node scripts/seed-role-entries.mjs --import --dataset=staging

npm run seed:footer-data                            # write NDJSON only
node scripts/seed-footer-data.mjs --import          # write + import production
node scripts/seed-footer-data.mjs --import --dataset=staging
```

Re-running the import is idempotent because each document carries a stable
`_id`. The seed script uses `sanity dataset import --replace`, so re-runs
overwrite Studio edits — only run `--import` when intentionally resetting
the dataset. Editors may edit values in Studio afterward without conflict
as long as the seed script is not re-imported.
